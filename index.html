Working with me
How I Work
🧭
Engineering Leadership Style
I run lean, async-first teams with clear ownership and explicit decision records. I unblock engineers before standups, not during them. I treat architecture decisions as team conversations, not top-down mandates — every significant change gets an ADR so context survives beyond the sprint.
🏗️
Architecture Approach
Start with constraints, not frameworks. I map data flows before writing a line of code, ask where idempotency breaks before adding a queue, and design for observable failure first. High-scale distributed systems are mostly failure management — I build for the unhappy path and let the happy path take care of itself.
⚡
AI-Augmented Workflow
I use Claude Code agents and Cursor as engineering force multipliers — not to skip thinking, but to compress the time between thinking and shipping. Architecture reviews that used to take days now produce a first ADR draft in hours. Test coverage that would have waited until post-merge gets generated in the same PR. The work is still mine; the pace is different.

## Role
You are a senior distributed systems architect // Sets the frame
 
## Context
Kafka consumer: PaymentProcessor — 1M msg/day
Current state: partition 7 lag = 280k, concurrency = 3
Constraint: idempotency must be preserved across retries
 
## Task
Design a scaling approach that eliminates lag
without risking duplicate payment processing.
Consider partition rebalancing side effects.
 
## Chain of Thought
Think step by step: diagnose → root cause → fix → risks
 
## Format
ADR format: context · decision · consequences · rollback plan
